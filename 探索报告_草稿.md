# 1. 任务与目标

本次考核要求实现一个限制输入窗口的文本分类 Harness。评测系统先通过 `update(text, label)` 逐条输入带标签训练样本，再通过 `predict(text)` 输入无标签测试文本，要求 Harness 返回一个与训练集中 label 完全一致的字符串。模型权重不更新，所有“学习”都发生在 Harness 维护的外部状态中，包括记忆、索引、检索、上下文构造、输出解析和 fallback。

运行接口可以概括为：

| 接口 | 输入 | 输出 | Harness 职责 |
| --- | --- | --- | --- |
| `update(text, label)` | 训练文本与 label | 无 | 保存训练样本，维护 label allowlist，准备索引和 label memory |
| `predict(text)` | 测试文本 | 一个 label 字符串 | 路由任务类型，检索候选，构造预算内 prompt，调用 LLM，解析并校验输出 |

关键约束决定了设计方向：

- 只提交 `solution.py`，最终逻辑必须封装在 `MyHarness` 内。
- 禁止读写文件，不能依赖本地缓存、共享数据文件或 FAQ 文件。
- 测试 label 不可见，不能硬编码公开 DEV 测试答案或错误样本。
- 单次 prompt 输入 token 必须小于 2048，因此必须主动做候选裁剪和预算控制。
- 输出必须 exact match，因此需要训练 label allowlist、输出解析和 fallback。
- 私有集包含 DEV-like 分类、OOD 分类、复杂自然语言选择题和少量 prompt injection，不能只针对客服 DEV 集写规则。
- 正式模型为 Qwen3-8B Instruct 非思考模式，小模型更适合短、明确、受控的 prompt。

本报告用五类指标评价策略：

| 维度 | 关注点 |
| --- | --- |
| 准确率 | DEV、共享 MCQ/OOD、临时 label 格式变体上的 exact-match accuracy |
| 稳定性 | 多轮波动、API 错误、非法输出、并发下是否异常 |
| 泛化性 | 是否适用于 DEV-like、OOD、MCQ、不同 label 格式和 injection 文本 |
| 预算控制 | prompt 是否主动低于 2048，是否依赖自动截断 |
| 提交合规性 | 是否只依赖 `solution.py`，是否无文件读写、无 API key、无测试标签泄漏 |

初始 baseline 是一个增强版检索分类 Harness：`update()` 保存训练样本；`predict()` 用 BM25、字符 n-gram 和 label 名称相似度检索候选 label，选择少量相关 examples 构造 prompt，让 Qwen3-8B 在受控候选集合中返回一个 label，并通过 allowlist parser 和本地 fallback 保证输出合法。它被选为起点，是因为初始百炼单轮 DEV 为 80.5%，prompt/条约 451 token，无 API 错误、无截断 warning，基础链路稳定且没有明显合规风险。

# 2. 探索路径概览

整体探索不是按日期线性堆实验，而是按“问题发现 -> 设计假设 -> 实验验证 -> 失败分析 -> 策略收敛”推进。先建立稳定 baseline，再通过错误分型判断瓶颈；随后验证候选扩张、prompt 加料、局部对比和 reranking 等直觉方案是否有效。多数“更多上下文”的方案被证伪后，探索转向更结构化的 label memory 和任务路由；最后用共享数据检查泛化风险，并做提交前合规与预算审计。

| 阶段 | 触发问题 | 尝试方向 | 观察结果 | 决策 |
| --- | --- | --- | --- | --- |
| Baseline | 需要稳定起点 | 检索 + few-shot + token 预算 + parser/fallback | DEV 单轮 80.5%，prompt/条 451，无 API 错误 | 保留为实验起点 |
| 错误分型 | 不清楚失败来自哪里 | 追踪候选命中、LLM 错判、parser、API、token | `candidate_hit_llm_wrong=74`，`candidate_miss=31`，`api_error=0` | 不优先修调用链路，转向候选与判别 |
| 候选/Prompt 加料 | 假设更多候选或规则能提升 | 扩候选、扩 examples、全局 prompt 规则、检索 override | `top16 + 12 examples` 79.6%；规则加法 78.7%；override 无收益 | 停止全局加料 |
| 局部判别与 Listwise | 假设更精细对比能减少误判 | label hints、contrast、example order、listwise ranking | 78.5%、79.2%、78.5%；listwise 77.7/77.4/68.6 | 不采用，说明复杂判别话术更不稳 |
| Label Memory | 需要在不堆 examples 的情况下表达 label 概念 | label prototype / concept memory 多形态实验 | 替代 examples 78.1%；inline cues 78.9%；cards + 4 examples 80.891% 且 4 轮稳定 | 正向组合落入 `solution.py` |
| 共享数据泛化审核 | 需要评估 MCQ/OOD 和外部数据风险 | 独立验证 MCQ/OOD；测试共享 train 合并 | MCQ 73.2%，OOD 78.0%；直接合并 DEV 降到 80.1% | 共享数据只做验证，不并入提交 Harness |
| MCQ 路由与 label 映射 | MCQ label 无语义，且正式 label 格式不确定 | 无示例 MCQ prompt；非单字符 label marker 映射 | MCQ 75.5%，DEV 80.9%；`Option A/B/C/D` 78.6% | 隔离式 MCQ 路由和精确映射落代码 |
| Auto-Tuning 验证 | 固定参数可能不适合私有集 | train-only LOO；两个保守 preset API 验证 | LOO 覆盖提升，但 API 为 80.148% / 79.221%，低于主线 | 不默认落代码，只保留诊断价值 |
| 提交前风险审计 | 确认最终代码可提交 | 静态扫描、预算审计、并发压力 | 无非法返回、无超预算、无合规问题；DEV max 1496，OOD max 1463，MCQ max 695 | 当前代码作为提交候选 |

这个过程体现了两次关键收敛：第一，错误分型证明主要问题不是 API、parser 或 token，而是候选内判别；第二，多轮负向实验说明“更多上下文、更多规则、更多候选”不稳定，最终应保留低噪声、可回退、可泛化的机制。

# 3. 不同 Harness 策略尝试

## A. Baseline 与错误诊断

**设计动机**

先构建一个满足提交边界的通用 Harness，而不是直接做复杂投票、子 agent 或针对 DEV 的规则。Baseline 需要同时具备记忆管理、动态检索、候选裁剪、token 预算、输出解析和 fallback。

**具体尝试**

- 增强版检索分类 Harness：BM25、字符 n-gram、label-name similarity 混合检索。
- 候选 label 裁剪后构造 few-shot prompt，LLM 只在候选集合内选择。
- 输出经过 exact/normalized parser，非法输出回退到检索 top-1。
- 低并发百炼单轮验证，并进一步做调用链路错误分型。
- 未改代码的 4 轮稳定性确认。

**实验结果**

- Baseline 单轮 DEV 80.5%，prompt/条 451，completion/条 3.9，无 API 错误、无截断。
- 错误分型：`candidate_hit_llm_wrong=74`，`candidate_miss=31`，`api_error=0`，`parser_missed_gold=0`。
- 4 轮稳定性：80.3% / 80.3% / 80.5% / 80.3%，平均 80.4%。

**分析与决策**

Baseline 已经稳定，且失败主要来自候选命中后的模型误判和少量候选 miss，不是调用链路、parser 或 token 预算问题。因此后续探索不优先修 API 或输出解析，而是围绕候选召回、候选内判别和任务泛化展开。

## B. 候选与 Prompt 加料探索

**设计动机**

错误分型显示存在候选 miss，也存在候选命中后的 LLM 误判。一个直接假设是：扩大候选、增加 examples、加入判别规则或利用检索高置信覆盖，可能提高准确率。

**具体尝试**

- `top16 + 12 examples`：扩大候选和示例数量。
- Prompt 规则加法：加入更宽泛的自然语言判别规则。
- 检索高置信覆盖：当检索 top-1 分数足够高时尝试覆盖 LLM 输出。
- 局部提示、共享 label token contrast、示例顺序调整。

**实验结果**

- `top16 + 12 examples`：候选命中率从 94.2% 提升到 95.5%，gold 示例覆盖到 92.4%，但 DEV 单轮降到 79.6%。
- Prompt 规则加法：DEV 78.7%。
- 检索 override：无阈值组合超过 80.5% baseline。
- 局部 hints / contrast / example order：78.5%、79.2%、78.5%。

**分析与决策**

这些实验说明，离线召回或信息量提升不能直接转化为最终准确率。对 Qwen3-8B 来说，更多候选、更多 examples、更多规则可能让候选内判别更困难。最终决策是不采用这些加料方案，避免把 prompt 做得更长、更噪声化。

## C. Reranking 与 Label Memory

**设计动机**

在直接分类中，模型需要从多个候选中选择一个 label。一个备选思路是将任务改写成候选排序或重排；另一条思路是保留直接分类，但用更压缩的 label memory 表达类别概念。

**具体尝试**

- Listwise reranking：让 LLM 对候选 label 排序后返回 rank-1。
- Listwise 变体：同候选、更多候选少示例、候选-only。
- Label Prototype / Concept Memory：从训练样本代码生成 label cards。
- Label Memory 变体：替代 examples、inline cues + examples、更多候选 + memory cards + 4 examples。

**实验结果**

- Listwise 三个变体分别为 77.7%、77.4%、68.6%。
- Label Memory 替代 examples：78.1%。
- Inline cues + examples：78.9%。
- 更多候选 + memory cards + 4 examples：80.891%，4 轮确认均为 80.891%；落代码后 DEV 80.9%，prompt/条 1074，无截断。

**分析与决策**

Listwise 失败说明排序任务并不比直接分类更容易，尤其对小模型而言，复杂输出格式和枚举比较会增加不稳定性。Label Memory 的关键发现是：prototype 不能替代原始 examples，但可以作为压缩后的类别概念，与少量 evidence examples 组合，支撑更多候选而不显著增加噪声。因此最终采用“label memory cards + 少量 evidence examples”的组合。

## D. 泛化任务与数据侧探索

**设计动机**

正式私有集包含 OOD 和复杂自然语言选择题，不能只看 DEV 客服意图分类。用户提供的共享数据可用于审核泛化风险，但不能直接假设可作为训练增强。

**具体尝试**

- 审核 `data.zip` 中的共享 MCQ/OOD 数据，检查格式、重复和 split 重叠。
- 用当前 Harness 独立验证 MCQ/OOD。
- 将共享 train 直接合并到 DEV train，观察数据增强风险。
- 只合并同名 `exchange_rate` 的 3 条外部样本。
- 对 MCQ 做隔离式无示例路由。
- 支持非单字符 MCQ label 的 marker 到原始 label 映射。

**实验结果**

- 共享 MCQ 独立任务：73.2%；OOD 独立任务：78.0%。
- 直接合并共享 train 后 DEV 降到 80.1%，纯外部 label 出现在 468/539 个 DEV 样本候选中，gold 候选命中下降。
- 仅同名 `exchange_rate` 3 条增强后 DEV 到 81.1%，但收益约 1 个样本且证据薄。
- 隔离式 MCQ 路由：MCQ 75.5%，prompt/条从 628 降到 189；DEV 保持 80.9%。
- 泛化 MCQ label 映射：原始 MCQ 保持 75.5%；临时 `Option A/B/C/D` label 格式达到 78.6%；DEV 保持 80.9%。

**分析与决策**

共享数据适合作为外部分布验证集，但不适合作为提交 Harness 的普通训练增强。直接合并会污染 label allowlist 和候选空间；同名 label 小样本增强虽然单轮到 81.1%，但有外部数据依赖、硬编码和泛化风险，因此没有采用。MCQ 的结论更明确：选项 label 本身无类别语义，训练 examples 反而干扰；隔离式无示例 prompt 和精确 label 映射更适合选择题，且不会影响普通 DEV 路径。

## E. Auto-Tuning 与提交前审计

**设计动机**

私有任务的 label 数量和分布可能与 DEV 不同，固定候选数、example 数和 label card 长度未必最优。一个低风险思路是在训练集内部做 leave-one-out，观察不同 preset 的候选覆盖和 token 预算。

**具体尝试**

- Train-only Auto-Tuning LOO：不调用 LLM，不看 test label，只用 train 内 leave-one-out 比较 preset。
- 评估 `recall24_e2_compact`、`recall20_e2_compact`、`compact_current` 等候选。
- 对两个保守候选做 API 单轮验证。
- 做提交前风险审计：静态扫描、无 API 预算审计、合成压力样本、50 并发 fake-LLM 压测。

**实验结果**

- DEV train LOO 中，当前 selected hit 为 86.6%，`recall24_e2_compact` 为 91.3%，`recall20_e2_compact` 为 88.7%，`compact_current` 仍为 86.6% 但 token 更低。
- OOD train LOO 中，当前 91.9%，`recall24_e2_compact` 94.1%，`recall20_e2_compact` 93.5%。
- API 验证：`compact_current` DEV 80.148%，`recall20_e2_compact` DEV 79.221%，均低于当前 80.9%。
- 提交前审计：DEV max prompt 1496，OOD max 1463，MCQ max 695；无非法返回、无超预算、无合规问题。

**分析与决策**

LOO 是有价值的结构诊断，但它只是候选覆盖代理指标，不能证明 LLM 最终会选对。API 验证显示 token 降低和候选覆盖提升没有转化为准确率，因此 Auto-Tuning 不默认落代码，只保留为 token 压力预案。提交前审计确认当前 `solution.py` 满足预算、安全和合规边界，适合作为提交候选。

## 失败方向分析表

| 失败方向 | 表面动机 | 实际现象 | 得到的结论 | 对最终方案的影响 |
| --- | --- | --- | --- | --- |
| 更多候选和 examples | 提高 gold 召回，让模型看到更多证据 | 候选命中率提升，但 DEV 降到 79.6% | 召回不是唯一瓶颈；候选内判别会被更多相似 label 干扰 | 最终只适度扩大候选，并用 label memory 压缩类别概念 |
| Prompt 规则加法 | 用规则帮助模型判别 | DEV 降到 78.7% | 小模型会被宽泛规则噪声干扰 | Prompt 保持短、直接、任务边界清晰 |
| 检索 top1 override | 用高置信检索纠正 LLM | 无阈值组合超过基线 | 检索分数不等于语义正确 | 检索只做候选生成和非法输出 fallback |
| 局部 hints / contrast | 给相近 label 额外差异提示 | 78.5% / 79.2% / 78.5% | 轻量局部信息未稳定提升 | 不做手写 confusion 规则，避免 DEV 领域过拟合 |
| Listwise reranking | 让模型排序候选，降低开放分类难度 | 同候选 ranking 也降到 77.7% | ranking 任务比直接分类更不稳 | 保留直接分类，不改成排序任务 |
| 共享数据直接合并 | 增加训练信息，提升泛化 | DEV 降到 80.1%，外部 label 污染候选 | 不同任务 label 空间混合会污染 allowlist | 共享数据只做外部分布验证，不进入提交 Harness |
| Auto-Tuning 默认化 | 用 train LOO 自动选更优 preset | LOO 指标提升，但 API 低于主线 | 离线代理指标不能直接替代 API 验证 | 不把 LOO 最优 preset 写死，只保留为诊断和预案 |

这些失败方向共同说明：更多上下文、更多规则、更多候选并不一定更好。对 Qwen3-8B 和 2048 token 限制而言，低噪声、强约束、可回退的 Harness 比复杂 prompt 或多轮机制更重要。

# 5. 最终方案与总结

最终方案一句话：

> 代码侧负责外部记忆、混合检索、候选裁剪、label memory cards、任务路由、token 预算、输出解析和 fallback；Qwen3-8B 只在受控候选集合内做一次最终语义选择。

## 最终 Harness 架构图

```mermaid
flowchart LR
    U[update(text, label)] --> M[External Memory]
    M --> I[Lazy Index]
    I --> A[Label Allowlist]
    I --> C[Label Memory Cards]

    P[predict(text)] --> EI[Ensure Index]
    EI --> R{Task Router}

    R -->|choice-like labels/text| MQ[MCQ Route]
    MQ --> MP[MCQ Prompt<br/>no examples]
    MP --> MM[Marker Mapping<br/>A/B/1 -> original label]
    MM --> PF[Parser + Allowlist + Fallback]

    R -->|generic classification| GR[Generic Classification Route]
    GR --> CR[Candidate Retrieval<br/>BM25 + char n-gram + label similarity]
    CR --> CB[Context Builder<br/>labels + cards + evidence examples]
    CB --> BC[Budget Controller<br/>keep prompt < 2048]
    BC --> LLM[Single LLM Call<br/>Qwen3-8B]
    LLM --> PF

    PF --> FL[Final Exact-match Label]

    R -. fallback if MCQ not reliable .-> GR
```

## 最终 Harness 模块表

| 模块 | 采用机制 | 作用 |
| --- | --- | --- |
| 外部记忆 | `update()` 保存训练样本、label 顺序和 allowlist | 保证输出空间来自当前任务训练流 |
| 混合检索 | BM25 + char n-gram + label-name similarity | 在 2048 token 内缩小候选 label 空间 |
| Label Memory | 代码生成 label cards + 少量 evidence examples | 用压缩类别概念支撑更多候选，同时保留原始样本证据 |
| 任务路由 | 普通分类、小 label、MCQ、injection-aware 分支 | 针对不同任务形态选择不同上下文策略 |
| MCQ 路由 | 无示例选择题 prompt + marker 到原 label 映射 | 降低选择题噪声，支持非单字符 label exact-match |
| 预算控制 | 调用前削减 examples、候选和长文本 | 主动保证 prompt 小于 2048，不依赖自动截断 |
| 输出控制 | parser + allowlist guardrail + retrieval fallback | API 异常或非法输出时仍返回训练 label |

最终方案不是把所有单轮高分或有用信号都堆上去，而是保留机制通用、预算安全、调用稳定、可解释且符合提交约束的部分。最典型的例子是：只合并同名 `exchange_rate` 的 3 条外部样本后，DEV 单轮到 81.1%，但该收益只有约 1 个样本，证据薄，且依赖外部共享文件；正式提交不能读外部文件，硬编码也会削弱 Harness 自适应叙事并增加复核风险。因此该变体没有进入最终方案。

已知局限性：

- 私有格式未知：正式 MCQ label 或题干格式可能不同于共享样本；当前只在能建立可靠 marker 映射时触发 MCQ，否则回退通用分类。
- 极端长文本和小 label 任务：小 label 且长文本时 prompt 可能接近预算上限；当前通过预算循环削减 examples、候选和文本，审计中 8 类长文本 max 1916。
- 小模型候选内判别能力有限：错误分型显示候选命中后仍有较多 LLM 误判；Label Memory 能缓解但不能完全消除。
- 检索不是语义真值：BM25/char/label-name 分数高不一定正确，因此检索不覆盖合法 LLM 输出。
- MCQ 路由有触发边界：如果 label 或题干格式无法可靠映射，则不强行触发，以避免误伤普通分类。
- 不默认多次调用：self-consistency 或 pairwise 可能对少量难例有帮助，但会增加延迟、限流和随机性风险。

方法论总结：

1. 负向实验也是结论。扩候选、prompt 规则、检索 override、局部 hints、Listwise 和数据合并的失败，帮助我们缩小搜索空间，避免继续沿高噪声方向调参。
2. 离线代理指标不能直接替代 API 验证。候选命中率和 train-only LOO selected hit 有诊断价值，但不能证明 Qwen3-8B 最终会选对。
3. Harness 的关键是用确定性代码控制搜索空间和输出空间。代码负责记忆、检索、路由、预算和校验，LLM 只做一次受控语义判断。
4. 最终策略必须同时满足准确率、泛化性、预算安全、稳定性和提交合规。单轮 DEV 的偶然提升不足以作为落代码依据。

最终结论是：当前提交候选不是最复杂的 Harness，而是经过正负实验筛选后的低噪声单调用 Harness。它保留了被验证有效的通用机制，拒绝了上下文过重、外部数据依赖或容易过拟合 DEV 的变体，更符合私有集 OOD/MCQ/injection 混合评测和 2048 token 限制下的工程目标。
