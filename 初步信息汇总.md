## 3. 考核硬约束

本次考核的关键限制：

- 只能提交一个 Python 文件：`solution.py`。
- 只应修改 `MyHarness` 类内部逻辑。
- 允许使用 Python 标准库、`numpy` 和 `harness_base`。
- 禁止读写任何文件。
- 禁止通过任何途径获取测试集标签。
- 单次 `call_llm` 的 prompt token 数必须小于 `max_prompt_tokens`，默认上限为 2048。
- Prompt 超过预算时，`run.py` 会从尾部截断并打印 warning，因此应主动预算控制，而不是依赖自动截断。
- 总 token 没有明确的全局限制，官方补充说明只限制单次调用的输入。
- 单个标签最大长度随任务不同而不同，但官方保证小于 50 字符。
- 正式评测使用 Qwen3-8B Instruct，且不开思考模式。
- 最终测试环境补充版本：`numpy==2.4.4`、`openai==2.32.0`、`transformers==5.6.2`。
- 私有评测包含多类任务：
  - 与 DEV 集标签一致但文本不同的分类任务。
  - OOD 文本分类任务，领域、标签、标签数量可能完全不同。
  - 复杂自然语言选择题任务，标签可能是 `A/B/C/D`。
  - 少量 prompt injection 样本。
- 最终成绩以私有集加权平均准确率为主，客观得分占 80%。
- 主观评分占 20%，关注 Harness 设计的创新性、合理性、可解释性等。

## 4. 模型与 API 背景

本次方案面向的是 Qwen3-8B，小参数量、小上下文的模型环境。

外部信息要点：

- Qwen3-8B 官方模型卡说明它是 8.2B 参数模型，支持 thinking / non-thinking 切换；考核代码通过 `enable_thinking=False` 禁用思考模式。
  - 来源：[Qwen/Qwen3-8B](https://huggingface.co/Qwen/Qwen3-8B)
- SiliconFlow 上的模型 ID 通常是 `Qwen/Qwen3-8B`，OpenAI-compatible base URL 通常是 `https://api.siliconflow.com/v1`。
  - 来源：[SiliconFlow Qwen3-8B](https://www.siliconflow.com/models/qwen3-8b)
- 官方补充说明：使用哪家 Qwen API 不重要，只要模型是 Qwen3-8B Instruct、关闭思考模式，且 API 格式为 OpenAI-compatible 即可。
- Harness 和模型调用适配需要根据具体 API 厂商调整；不同 infra 的适配方式可能不同。
- 当前本地百炼接入使用 `DASHSCOPE_API_KEY`、`DASHSCOPE_BASE_URL`、`DASHSCOPE_MODEL` 等环境变量；Qwen3 非思考模式参数采用 `extra_body={"enable_thinking": False}`。
- 由于 Qwen3-8B 参数量较小，不能假设它能稳定处理长 prompt、多轮复杂推理、自由格式输出或开放标签选择。
- 因此 Harness 应承担更多确定性工程工作：记忆管理、检索、候选裁剪、token 预算、输出解析、纠错和防注入。

## 5. 外部研究与工程参考

几个对当前方案有直接影响的公开结论：

- Retrieved Demonstrations / RetICL 的核心结论是：ICL 对示例选择非常敏感，按测试输入动态检索示例通常优于固定 few-shot。
  - 来源：[TMLR 2024 RetICL Survey](https://mlanthology.org/tmlr/2024/luo2024tmlr-incontext/)
- 示例选择存在不稳定性，label coverage、样本顺序、样本质量都会影响 ICL。
  - 来源：[ACL Anthology: Active Example Selection for In-Context Learning](https://aclanthology.org/2022.emnlp-main.622/)
- 长上下文里相关信息放在中间会被弱化，模型更容易使用开头和结尾的信息。虽然本题限制 2048 token，但仍支持“少放、精选、把关键约束和候选标签放显眼位置”的设计。
  - 来源：[Lost in the Middle](https://direct.mit.edu/tacl/article/doi/10.1162/tacl_a_00638/119630/Lost-in-the-Middle-How-Language-Models-Use-Long)
- Prompt injection 的根因是模型难以天然区分“指令”和“数据”，因此应在 prompt 结构和输出验证层做边界控制。
  - 来源：[OWASP Prompt Injection](https://owasp.org/www-community/attacks/PromptInjection)
- 广泛 Harness / Agent 工程中，Anthropic 推荐优先使用简单、可组合、可控的 workflow，而不是一开始就堆复杂 agent。
  - 来源：[Anthropic Building Effective Agents](https://www.anthropic.com/engineering/building-effective-agents)
- OpenAI Agents SDK 中的 handoff 本质上是把“转交给某个 agent”表示成工具调用，适合任务边界明确、需要多角色接管的场景。
  - 来源：[OpenAI Agents SDK Handoffs](https://openai.github.io/openai-agents-python/handoffs/)
- Guardrails 思路强调输入、输出和工具调用都需要边界校验。在本考核中，最重要的是输出合法性校验和 prompt injection 防护。
  - 来源：[OpenAI Agents SDK Guardrails](https://openai.github.io/openai-agents-python/guardrails/)

## 6. 核心叙事

本次方案不应是“把所有训练集塞进 prompt”，而应是：

> 用确定性代码做记忆、检索、候选裁剪、token 预算和输出校验；只把最相关、最短、最有区分度的信息交给 Qwen3-8B 做最终语义判断。LLM 负责小范围决策，Harness 负责控制问题规模。

这一路线更适合：

- Qwen3-8B 这种小参数量模型。
- 2048 token 的小上下文限制。
- 高并发 API 评测环境。
- exact match 分类任务。
- 含 OOD、选择题和 prompt injection 的私有评测集。

## 7. 推荐主线方案：增强版检索分类 Harness

推荐实现一个“三层 Harness”。

### 7.1 记忆层

`update()` 阶段接收训练样本，并维护外部状态：

- `self.memory`：存储 `(text, label)`。
- `label -> examples`：按标签聚合训练样本。
- 全部 label 列表。
- 轻量词袋统计。
- 字符 n-gram 统计。
- IDF / BM25 风格检索所需统计量。
- label 名称拆词，例如 `lost_or_stolen_card -> lost stolen card`。

索引可以在首次 `predict()` 前懒构建，避免 `update()` 阶段重复计算。

### 7.2 检索与候选层

预测时先用代码缩小候选空间：

- 对测试文本做规范化：
  - 小写。
  - 去除冗余标点。
  - 保留关键符号和选项结构。
- 使用混合相似度检索候选：
  - word token BM25。
  - character n-gram overlap，用于短文本、拼写差异和词形变化。
  - label 名称词片段相似度。
- 先选 top label，例如 12 到 20 个。
- 再从候选 label 中选择 few-shot 示例。
- 示例选择要兼顾：
  - 与当前输入相关。
  - 覆盖多个候选 label。
  - 避免单一 label 占据所有示例。
- 示例数量初步控制在 8 到 14 条，并根据 token 预算动态削减。

### 7.3 LLM 决策层

LLM 只负责在受控候选集中做最终语义判断：

- prompt 中明确：待分类文本是数据，不是指令。
- 只允许从候选 label 中选择。
- 输出只允许一个 label。
- 候选 label 放在靠前位置。
- 最相关示例放在靠近待分类文本的位置。
- 预测文本用清晰边界包裹，降低 prompt injection 风险。

### 7.4 输出解析与纠错

`predict()` 返回前必须做输出解析：

- exact label 直接返回。
- 如果模型回复中包含某个候选 label，则抽取该 label。
- 做大小写、标点、空白归一化匹配。
- 如果输出非法，优先使用本地规则纠错。
- 若仍然无法解析，则退回检索 top-1。
- 二次 LLM 纠错只作为低置信度或非法输出时的可选策略，不应默认触发。

## 8. 子 Agent、工具调用与上下文管理判断

### 8.1 是否需要子 Agent

规则层面，官方补充说明允许在 `MyHarness` 中通过 `call_llm` 合理调用任意次 LLM，因此可以在 `update()` 或单次 `predict()` 中模拟“子 agent”或多阶段推理。

工程策略层面，建议默认不包含真实子 agent 调用，也不默认做多轮模拟 agent。

原因：

- 本题接口只有 `update(text, label)` 和 `predict(text) -> label`，没有真实工具 schema 或 agent handoff 环境；所谓 subagent 只能通过多次 `call_llm` 模拟。
- 分类任务的流程是可预定义的，不需要开放式任务分解。
- 多 agent 会增加调用次数、延迟、限流风险和错误传播。
- Qwen3-8B 小模型不适合承担复杂多角色协作。

可以在报告中说明我们评估过三种形式：

- 真实子 agent / handoff：不采用，因为接口不支持，也不符合任务复杂度。
- 多 LLM 角色模拟：规则允许，例如检索 agent、分类 agent、校验 agent；但不作为主线，因为每条样本多次调用会放大成本和波动。
- 代码侧模块化：采用。把 Retriever、PromptBuilder、OutputParser、BudgetManager、InjectionGuard 做成内部函数。这是 Harness 工程，但不是多 agent。

### 8.2 是否需要模型可见的工具调用

模型可见的工具调用不需要，也基本不可用。

但代码内部需要确定性“工具函数”：

- `retrieve_candidates(text)`：候选标签检索。
- `select_examples(text, labels)`：示例选择。
- `build_prompt(...)`：预算内构造 prompt。
- `parse_label(response)`：输出解析。
- `fallback_label(text)`：非法输出时回退。
- `is_injection_like(text)`：识别疑似注入文本并强化边界提示。

这些函数不暴露给 LLM，由 Harness 代码直接执行，比让模型决定是否调用工具更稳定。

### 8.3 上下文如何管理

上下文不应是长对话历史，而应是“外部记忆 + 动态检索上下文”。

推荐结构：

- `update()` 阶段积累外部记忆和索引。
- `predict()` 阶段临时构造最小充分上下文。
- prompt 中只放与当前输入最相关的候选标签和示例。
- 不保留测试样本之间的对话历史，避免污染与 token 浪费。
- 不依赖自动截断，主动用 `count_messages_tokens` 控制预算。

## 9. 并发问题

讨论中提到的“并发”指的是：`run.py` 会同时对多条测试样本调用 `harness.predict()`，而不是一条一条顺序执行。

代码中对应：

```python
ThreadPoolExecutor(max_workers=args.workers)
```

默认 `--workers 20`，意思是最多同时运行 20 个 `predict()`。如果每个 `predict()` 调用一次 LLM，就可能同时向硅基流动发起最多 20 个 API 请求。

需要关注并发的原因：

- SiliconFlow 这类 API 服务通常会限制并发请求数、QPS、RPM 或 TPM。
- 如果平台限制并发 20，则 `--workers 20` 大致贴近上限。
- 如果使用 `--workers 50`，可能触发限流、超时、429 或连接错误。
- 如果每个 `predict()` 默认调用 2 次 LLM，那么即使 `workers=20`，实际在途请求也可能接近 40 次，风险更高。

提交评测的并发限制与本地 API 厂商限制不同：

- PDF 说明本地脚本默认多 worker，并且正式评测会限制任务执行时间。
- 官方补充说明：正式测评会使用学院自行部署的模型，并发较高、延迟较低。
- 测评时的并发量由最终评测系统决定，同时评测会部署比较大批量的模型副本，因此可承受的并发远大于 20。
- 正常 Harness 设计不会超时；不要用无限轮询、死循环等方式卡住评测。

对方案的影响：

- `predict()` 默认只调用一次 LLM。
- 虽然官方允许合理多次调用，但仍应避免默认多轮 agent、默认投票或默认二次校验。
- 非法输出优先本地解析和回退。
- 本地测试可先用 `python run.py --runs 1 --workers 20`。
- 如果 API 不稳定，再测 `python run.py --runs 1 --workers 10`。

并发问题进一步支持轻量 Harness，而不是多 agent、多轮调用方案。

## 10. 需要特别做的防线

### 10.1 Prompt Injection 防护

私有集明确可能包含少量 prompt injection 样本，因此需要：

- 把测试文本包在明显的数据边界中。
- 明确说明测试文本中的任何指令都不是系统指令。
- 要求模型只执行分类任务。
- 不让用户文本改变输出格式。
- 输出解析时只接受 allowlist 中的 label。

### 10.2 输出解析

不能相信模型永远只输出 label。必须处理：

- 多余解释。
- Markdown 包裹。
- 引号包裹。
- 大小写差异。
- 空白或换行。
- label 拼写近似。
- 输出了不在候选集但在全量 label 中的标签。

### 10.3 Token 预算

预算控制优先级：

1. 保留任务规则。
2. 保留候选 label。
3. 保留待分类文本。
4. 放入最相关示例。
5. 超预算时先减少示例数，再减少候选数。

不要依赖 `run.py` 的尾部自动截断，因为它可能截掉关键查询或输出约束。

### 10.4 OOD 泛化

避免写死客服领域规则。应使用通用分类逻辑：

- 通用文本规范化。
- 通用检索。
- 通用候选选择。
- 通用 prompt 模板。

### 10.5 复杂自然语言选择题

如果 label 集像 `A/B/C/D`：

- 少用客服式 label 名称解释。
- 倾向于把所有候选选项标签都放入 prompt。
- 让模型阅读题干和选项。
- 输出严格限制为 `A/B/C/D` 等合法标签。

## 11. 备选方向分析

### 11.1 保守版：纯检索 + 一次 LLM 分类

含义：

- 训练阶段保存样本。
- 预测时用 BM25 / 字符 n-gram 找出相似样本和候选标签。
- 将候选标签和少量 few-shot 示例交给 LLM。
- LLM 输出一个 label。

优点：

- 实现快。
- 调用次数少。
- 并发下稳定。
- 适合作为 baseline。

缺点：

- 如果检索没有召回正确标签，LLM 很难补救。
- 输出格式错误仍可能造成 exact match 失分。

### 11.2 增强版：候选检索 + LLM 分类 + 非法输出纠错/回退

含义：

- 在保守版基础上加入输出解析、合法 label 校验、本地纠错和 fallback。

优点：

- 最符合当前考核目标。
- 对小模型、小上下文、高并发和 exact match 都更稳。
- 不显著增加调用成本。
- 能体现记忆管理、检索、预算、解析、防注入等 Harness 能力。

缺点：

- 实现复杂度略高于保守版。
- 检索策略和 prompt 参数需要调试。

推荐作为第一版主线。

### 11.3 激进版：每个 label 先生成语义描述，再做描述级检索

含义：

- 为每个 label 维护语义描述，例如 `card_swallowed` 描述为“银行卡被 ATM 吞掉，需要取回”。
- 预测时不仅匹配原始样本，也匹配 label 描述。

优点：

- 可能提升 OOD 或标签名抽象任务。
- label 描述可作为压缩记忆，节省部分示例 token。

缺点：

- 生成描述需要额外 LLM 调用。
- 描述质量不稳定。
- 正式任务标签数量和类型未知。
- 会增加成本、时间和并发风险。

不适合作为第一版，可以作为报告中的后续探索方向。

### 11.4 纯传统分类器

含义：

- 不使用 LLM 或极少使用 LLM。
- 用词袋、相似度、kNN、朴素贝叶斯等传统方法预测 label。

优点：

- 快。
- 稳定。
- 不依赖 API 输出格式。
- 在 DEV 客服意图分类上可能表现不错。

缺点：

- 对 OOD 和复杂选择题泛化风险大。
- 如果标签是 `A/B/C/D`，传统相似度可能无法真正理解题目和选项。
- 主观评分中 Harness 设计合理性和 LLM 利用程度可能不足。

不建议作为主线，但可以作为 fallback 或对照 baseline。

## 12. 其他可借鉴的 Harness 方案

### 12.1 任务路由 Harness

借鉴 agent workflow 中的 routing 思路：先判断当前任务类型，再选择不同 prompt 和检索策略。

可路由类型：

- label 是 `A/B/C/D`：按选择题处理。
- label 是语义短语：按普通文本分类处理。
- label 数量很少：直接给全量 label。
- label 数量很多：先检索候选，再分类。
- 输入疑似 injection：使用更强的数据边界 prompt。

优点：

- 泛化好。
- 能适配私有集里的 OOD 和选择题。
- 实现成本可控。

风险：

- 路由规则过复杂会引入误判。

建议加入主线，作为轻量模块。

### 12.2 Listwise Reranking Harness

流程：

```text
query -> 检索 top-K labels -> LLM 重新排序或选择最佳 label -> 取 rank-1
```

优点：

- LLM 任务更窄，只需比较候选标签。
- 对小模型相对友好。
- 输出更容易控制。

缺点：

- prompt 稍长。
- Qwen3-8B 对排序格式可能不如大模型稳定。

可简化为“候选解释 + 选最匹配 label”，作为增强方向。

### 12.3 Pairwise Tournament Harness

流程：

```text
A vs B -> winner
winner vs C -> winner
...
```

优点：

- 每次判断很简单。
- 适合小模型做局部比较。

缺点：

- 调用次数太多。
- K=8 时可能需要 7 次以上 LLM 调用。
- 不适合并发 20 和正式评测时间限制。

适合作为理论备选，不建议实现。

### 12.4 Self-Consistency Voting Harness

含义：

- 对同一样本使用不同示例顺序、不同候选子集或不同 prompt 表述跑多次。
- 最终投票决定 label。

优点：

- 能降低采样波动。
- 复杂选择题上可能有帮助。

缺点：

- 成本和调用次数成倍增加。
- 并发下更容易限流。

建议只做条件触发版，例如检索 top-1 和 LLM 输出冲突、或输出非法时才二次调用；不要默认投票。

### 12.5 Prompt Compiler / Auto-Tuning Harness

借鉴 DSPy 思路，把 prompt、示例数、候选数、检索权重当作可调参数。

轻量可行版本：

```text
训练集到齐后
-> 用 leave-one-out 测不同检索权重
-> 选择 BM25 / char n-gram / label-name 权重
-> 选择每个 prompt 放多少候选和示例
```

优点：

- Harness 能自适应任务。
- 工程叙事强。
- 如果只用代码做，不额外调用 LLM，成本可控。

缺点：

- 实现复杂度更高。
- 如果用 LLM 参与自动调 prompt，会太贵且不稳定。

可作为第二阶段增强。

### 12.6 Label Prototype / Concept Memory Harness

含义：

- 不为每个 label 手写或生成描述，而是从训练样本中压缩出 label prototype。
- 每个 label 存：
  - label 名称。
  - 代表样本。
  - 关键词。
  - 短语片段。
  - 容易混淆的相邻 label。

优点：

- 省 token。
- 适合 2048 token 限制。
- 比 LLM 生成 label 描述更稳定。

缺点：

- 压缩可能丢失细节。
- 每类只有 3 条样本时，prototype 质量有限。

可作为后续增强。

### 12.7 Confusion-Aware Harness

含义：

- 自动识别容易混淆的 label 组。
- 例如：
  - `pending_transfer`
  - `failed_transfer`
  - `declined_transfer`
  - `transfer_timing`
  - `transfer_not_received_by_recipient`

预测时如果候选落在混淆组内，就重点比较这一组。

优点：

- 对客服意图分类有效。
- 符合记忆管理和可解释性要求。

缺点：

- 如果手写客服领域规则，会影响 OOD 泛化。

建议通用化：用 label 名称和样本相似度自动发现混淆组，而不是手写业务规则。

### 12.8 Guardrail-First Harness

含义：

```text
输入规范化
-> injection-like 检测
-> 强边界 prompt
-> LLM 分类
-> label allowlist 解析
-> 非法输出本地回退
```

优点：

- 对私有集中的 prompt injection 有针对性。
- 能减少 exact match 格式失误。
- 可作为所有方案的基础层。

缺点：

- 单独不能显著提升语义分类能力，需要与检索和候选选择结合。

## 13. 推荐优先级

当前推荐的实施顺序：

1. 主线增强版 + 任务路由 + guardrail-first：最值得实现。
2. 加入轻量 prompt compiler：用训练集自适应检索权重和 prompt 参数。
3. 加入 confusion-aware：自动识别混淆 label 组。
4. 视本地结果再考虑 label prototype。
5. 不默认做 self-consistency，只在低置信度时触发。
6. 不做 pairwise tournament 和完整多 agent，成本和限流风险太高。

## 14. 当前建议决策

建议先实现“增强版 baseline”：

- 单 agent。
- 单次 LLM 调用为默认路径。
- 代码侧完成记忆、检索、候选裁剪、token 预算、防注入、输出解析与回退。
- 可加入轻量任务路由。
- 不默认做多 agent、多轮投票或 pairwise tournament。

本地验证优先观察：

- DEV 准确率。
- 非法输出率。
- 平均 prompt token。
- 是否出现 prompt truncated warning。
- API 超时 / 限流情况。

后续根据错误样本决定是否加入：

- 条件性二次纠错。
- label prototype。
- confusion-aware 分组。
- prompt 参数自适应。

## 15. 官方 FAQ / 补充要求整理

以下内容来自补充截图中的问答，应作为后续实现和报告撰写的约束参考。

### 15.1 API、模型与运行环境

- 使用哪家 Qwen API 不固定；模型和 Harness 的适配需要同时调整，不同 infra 的适配方式不同，需要考生自行确认 API 厂商。
- 只要模型是 Qwen3-8B Instruct、关闭思考模式，且 API 格式为 OpenAI-compatible 即可。
- 正式测评会使用学院自行部署的模型，并发比较高、延迟较低。
- 正常 Harness 设计不会超时；只有无限轮询、死循环、恶意卡住评测等异常行为才会导致超时或 0 分风险。
- 最终测试使用的依赖版本补充为：`numpy==2.4.4`、`openai==2.32.0`、`transformers==5.6.2`。
- 本环境不需要使用 PyTorch。
- 测试环境 Python 版本在截图中仅显示“见 74 行”，当前截图未展开具体版本，需要后续回查 FAQ 原表确认。

### 15.2 `call_llm`、`update()` 与多次调用

- `update()` 阶段可以调用模型 API，使用注入的 `call_llm` 即可。
- 可以在 `MyHarness` 中合理调用任意次 LLM，包括在单轮 `predict()` 中多次调用 `call_llm`。
- 单次任务如果是整个 test 集测完后一共调用约 1000 次，这属于合理范围。
- 如果是单次 `predict()` 内调用约 1000 次，则可能过多，存在超时和资源风险。
- 官方没有限制一轮多次调用时的总 tokens，只限制单次调用的输入。
- 因此从规则上看，多阶段推理、模拟 subagent、纠错调用是允许的；从工程稳定性看，仍建议默认一次调用，低置信度或非法输出时再追加调用。

### 15.3 Token、标签与数据格式

- 输入数据是否存在溢出 2048 token、需要压缩的情况，官方答复为“无法告知”。
- 单个标签的最大长度随任务不同而不同，但保证小于 50 字符。
- `llm_client.py` 中 `max_tokens: 8192` 限制的是模型最大输出 token 数，不是单次 prompt 输入 token 上限。
- prompt injection 样本中包含正常文本和对应的分类标签；也就是说，注入样本仍然是正常参与分类的样本，不应被简单丢弃。
- 根据 PDF，私有集包含多个不同任务，最终会对多个任务正确率进行加权平均计算。

### 15.4 允许与不允许的代码行为

- `MyHarness` 类中除了 `update()` 和 `predict()`，可以为了代码整洁自行新增函数或辅助方法，但必须保证实现 `update` 和 `predict` 两个方法。
- 不允许读写文件。
- 只能使用 `MyHarness` 中注入的 `call_llm`、`count_tokens`、`count_messages_tokens` 方法。
- 如果想使用 `llm_client.py` 中其他函数，例如 `_load_tokenizer`，则需要在文件前面 `import` 该文件，这本身不被允许。
- 是否允许修改 `OPENAI_CONFIG`：官方答复为不能修改；如果本地修改，与最终测试参数也会不同，最终测试的参数按当前 `OPENAI_CONFIG` 所示。实际本地调试时仍需按考核说明配置 API 连接，但方案不应依赖修改评测侧参数。
- 关于是否可以修改 `extra_body`：截图答复为“见 84 行”，并补充本地测试时只有该参数格式正确才能正常调用 LLM。后续需要回查 FAQ 原表确认具体约束。

### 15.5 报告与个人信息

- 报告中可以标明姓名和报名号，也可以不标明；最终上传后会自动重命名。
- 自动重命名指的是提交文件名，方便考官辨认提交人。
- 如果愿意，也可以在 report 中写明个人信息。
- 报告中是否限制出现真实信息和内容，官方答复为不太确定这里的“真实信息和内容”具体指什么；最终报告由考官查看，做好必要的隐私保护即可。

### 15.6 对当前方案的影响

这些补充要求不会推翻当前主线，但会调整表述：

- 规则上允许合理多次 LLM 调用和模拟 subagent。
- 工程上仍不建议默认多 agent / 多轮投票，因为本地 API 可能有并发限制，且 Qwen3-8B 小模型多轮错误传播风险更高。
- 正式评测 infra 比 SiliconFlow 本地调试环境更强，因此“本地并发 20 限制”不应被误认为官方评测硬限制。
- `update()` 可调用 LLM，意味着可以探索训练阶段构建 label prototype 或任务摘要；但第一版仍建议先不做，以降低成本和不稳定性。
- 对 prompt injection 的处理不能简单过滤样本，而应把注入文本当作待分类数据，并通过边界提示和 allowlist 输出解析保证安全。
