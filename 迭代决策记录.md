# 迭代决策与效果记录

本文档用于记录每次改进、调参、验证或方向切换的原因和实际效果。它服务于后续复盘和探索报告，不替代 `BASELINE_IMPLEMENTATION_PLAN.md`。

## 维护规则

- 每次修改 `solution.py` 的策略、prompt、检索、token 预算、输出解析、fallback 或并发处理后，都要新增一条记录。
- 每次验证运行后，都要把命令、准确率、token、耗时、warning 和错误情况补到对应记录。
- 未运行验证时，实际效果必须写为“待验证”，不要用主观判断冒充实验结论。
- 如果方向被放弃，要写清楚放弃原因、观察到的问题和替代方案。
- 记录应保留旧结论，后续修正用新条目说明，不静默覆盖历史判断。

## 当前基线状态

### 2026-05-07 - 百炼低并发单轮验证

- 背景/触发：用户要求使用阿里百炼 API 重新跑一轮测试，并减少并发、等待时间和重试次数；同时检查当前 API 调用的网络情况。
- 决策/改动：将 `llm_client.py` 的百炼兼容接口配置改为可由环境变量覆盖；默认 endpoint 使用北京 `https://dashscope.aliyuncs.com/compatible-mode/v1`，默认模型为 `qwen3-8b`；新增 `LLM_RETRIES`、`LLM_RETRY_BASE_SLEEP`、`LLM_TIMEOUT`、`LLM_MAX_TOKENS` 配置；按百炼当前 OpenAI 兼容接口要求把 Qwen3 非思考模式参数改为 `extra_body={"enable_thinking": False}`。
- 原因：避免把 API Key 写入文件；本轮需要比默认 `workers=20`、`retries=2`、线性 1 秒退避更保守；百炼接口当前不接受旧的 `chat_template_kwargs.enable_thinking` 写法。
- 预期效果：降低限流和长等待风险，真实验证当前网络与模型调用是否稳定。
- 实际效果：三地端点均可达，未鉴权探测按预期返回 401；北京端点最快。北京真实模型调用成功，单次轻量调用总耗时约 0.999s。DEV 单轮测试完成，无 API 错误、无 prompt truncated warning；仅出现 `transformers` 的 PyTorch 缺失提示，不影响 tokenizer 计数。
- 验证命令：`DASHSCOPE_API_KEY=<redacted> DASHSCOPE_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1 DASHSCOPE_MODEL=qwen3-8b LLM_RETRIES=0 LLM_RETRY_BASE_SLEEP=0.2 LLM_TIMEOUT=20 LLM_MAX_TOKENS=32 python run.py --runs 1 --workers 4`
- 指标：准确率 80.5%；prompt/条 451 token；completion/条 3.9 token；总耗时 58.0s；单轮各轮结果 80.5%。
- 影响文件：`llm_client.py`、`迭代决策记录.md`。
- 下一步：若继续调分，优先分析错误分布或尝试更稳的候选召回；若继续压测，可在当前网络状态下逐步试 `workers=6` 或 `workers=8`，但仍建议保留低重试配置。
