# 迭代决策与效果记录

本文档用于记录每次改进、调参、验证或方向切换的原因和实际效果。它服务于后续复盘和探索报告，不替代 `BASELINE_IMPLEMENTATION_PLAN.md`。

## 核心原则

- 先保证可回退：每次修改前确认当前本地基线已推送到 GitHub；没有稳定远端基线时，不继续做可能影响准确率的改动。
- 只让确定性代码做确定的事：记忆管理、检索、候选裁剪、token 预算、输出解析、fallback 和防注入由 Harness 控制；LLM 只在受控候选集合内做一次语义判断。
- 不为了 DEV 分数堆复杂度：只有实验收益明确、机制通用、不会明显增加限流和 token 风险时，才考虑把变体落入 `solution.py`。
- 负向实验也算结论：单轮显著低于当前基线区间的变体不再追加 4 轮确认；只有超过基线或提交前稳定性检查才跑多轮。
- 不把 API key、临时脚本或本地诊断逻辑写进提交文件；命令记录必须脱敏。
- 输出必须可控：最终返回必须来自训练 label allowlist；prompt injection 文本只作为待分类数据，不作为指令。
- 避免私有集风险：优先保留通用 Harness 机制，避免手写 DEV 客服领域规则、过度拟合错误样本或用检索覆盖模型判断。
- 默认单次调用：不默认启用 self-consistency、pairwise tournament 或多 agent；额外调用只在低置信、非法输出或正式风险明确时作为条件触发备选。

## 主线推进逻辑

先建立通用稳定基线，因为提交文件需要的不只是 DEV 准确率，还包括 token 预算、输出合法性、防注入、并发安全和 fallback。基线单轮 80.5%，且没有 API 或截断问题，所以后续不应先修基础设施，而应先诊断错误来源。

调用链路错误分型显示主要错误不是调用失败，也不是 parser 或 token 超限，而是候选命中后模型仍选错，以及少量候选 miss。因此先验证“候选更多、示例更多、规则更多、检索更强势”是否能解决问题。

候选扩张与规则调参的结果说明，离线召回提升不能直接转化为线上准确率；对 Qwen3-8B 来说，更大的候选集、更宽泛的规则和检索 override 都会带来新误判。于是探索范围收敛到更局部的 contrast / confusion-aware 小实验，并且默认单轮验证。

局部提示与轻量混淆对比仍然降分，说明当前瓶颈不能靠简单增加 label hints、共享 token 对比或示例重排解决。继续沿这个方向细刷 DEV，收益不确定，过拟合和 prompt 噪声风险更高。

Listwise 候选重排说明，把同一候选集包装成 ranking 任务不会自动提升候选内判别。Label Prototype / Concept Memory 的后续探索则给出了一个更具体的正向组合：不是用 prototype 替代 examples，也不是给候选行加少量 cues，而是用代码生成的 label memory cards 支撑更多候选，同时保留少量原始 examples。这个组合把 prompt 从 451/条提高到约 1074/条，但仍在 2048 预算内，并在 4 轮确认中稳定达到 80.891%。因此当前主线从“纯 few-shot 检索分类”升级为“候选检索 + label memory cards + 少量 evidence examples”的单调用 Harness。

共享数据侧增强审核把探索从 DEV 内部调参转到外部分布风险检查。`data.zip` 中新增 MCQ 和 OOD 数据没有文本重复或格式问题，适合作为“选择题长文本”和“非银行多域意图”的泛化验证集；但它们的 label 空间与 DEV 不同，不能作为当前任务的普通训练增强。直接把共享 train 合并到 DEV 会扩大 allowlist 并污染候选集，实际单轮也从 80.9% 降到 80.1%。只合并同名 `exchange_rate` 的 3 条样本虽然单轮到 81.1%，但收益仅约 1 个样本，且来自共享外部样本，不符合提交时不读外部文件、不过度依赖特定共享数据的原则。因此共享数据的结论是“用于验证和写报告，不进入提交 Harness”。

隔离式 MCQ 路由是在共享数据审核后的针对性增强：不把外部 MCQ 数据并入训练记忆，而是把 MCQ 作为一种可识别任务形态处理。实验显示，选择题 label `A/B/C/D` 本身没有语义，当前通用 few-shot 分类会把训练 examples 当作类别证据引入噪声；专用无示例 prompt 直接让 LLM 阅读题干和选项，成本更低且更准。该分支只在 choice label set 和 choice-like text 同时满足时触发，失效后回退通用 Harness，因此不会影响普通 DEV 路径。

FAQ 进一步说明，正式选择题任务是独立任务、单选、通用领域，但 label 不保证固定为 `A/B/C/D`，也不保证都是单字符；输出必须与训练集中 label 字符串完全一致。这使得单字母 MCQ 分支虽然安全，但覆盖不足。泛化 MCQ label 映射的策略是分层而不是替换：单字符 A-H 继续使用已验证有效的低噪声 prompt；非单字符选择题 label 只在能从 label 或题干选项中建立可靠映射时触发，并将模型输出的 `A`、`B`、`1` 等 marker 精确映射回训练集中原始 label 字符串。

训练集内 Auto-Tuning 离线 LOO 用来评估“提交时按当前任务 train 自适应”的可能性，而不是用公开 DEV test 刷分。结果显示，train 内候选覆盖确实可以通过 `recall20/24 + 更少 examples + 短 label cards` 提高，同时降低 prompt token；但这只说明 gold label 更常出现在候选集中，并不能证明 Qwen3-8B 在更大候选集里会选对。由于此前 `top16 + 12 examples` 已证明离线召回提升可能转化为线上降分，本方向暂不落代码，只作为后续少量 API 验证候选。

随后按用户要求对两个保守候选做官方 DEV API 单轮验证。`compact_current` 只压缩 label cards，准确率为 80.148%，比当前 80.9% 少 4 个正确样本；`recall20_e2_compact` 为 79.221%，比当前少 9 个正确样本。两者都没有 API 错误或截断，说明失败来自 Harness 策略本身，而不是调用链路。这个结果确认：Auto-Tuning 的 LOO 代理指标目前只能作为诊断信号，不能直接落入提交策略。

## 当前基线状态

- 决策/改动：先确认远端 `master` 已同步到 `4d060b5`；随后运行纯离线 leave-one-out 网格，不调用 LLM、不读取 test label、不修改 `solution.py`。每次从 train 中拿出 1 条作伪测试，其余样本进入 `update()`，评估 gold label 是否进入候选、prompt token、候选数和 example 数。比较少量通用 preset：当前配置、短 label card、候选 20/24 + 2 examples、6 examples、BM25/char/label 权重偏置、保守 12 候选。
- 原因：FAQ 显示 OOD 权重最高且单任务 label 可到 200，固定参数可能不是所有私有任务最优；但公开 DEV test 不能作为调参目标。Train-only LOO 可以作为更低风险的结构诊断，前提是只看训练集内部代理指标，并且不把离线最优直接当成最终准确率最优。
- 预期效果：如果当前固定参数在 train 内有明显候选覆盖短板，LOO 会显示某些通用 preset 在 DEV/OOD train 上都提升 selected candidate hit，且不会触发 2048 token 风险。若收益只来自显著增大候选集，则应保持谨慎。
- 实际效果：DEV train LOO 中，当前配置 selected candidate hit 为 86.6%，prompt p95 1308；`recall24_e2_compact` 为 91.3%，prompt p95 1144；`recall20_e2_compact` 为 88.7%，prompt p95 1004；`compact_current` selected hit 仍为 86.6%，prompt p95 降到 930。OOD train LOO 中，当前配置 selected candidate hit 为 91.9%，prompt p95 1291；`recall24_e2_compact` 为 94.1%，prompt p95 1047；`recall20_e2_compact` 为 93.5%，prompt p95 927；`compact_current` selected hit 仍为 91.9%，prompt p95 降到 861。MCQ train 结构检查 selected hit 为 100%，无需通过检索参数调优。
- 验证命令：`python -c <train_only_loo_grid_script>`，脚本只读取 `data/train_dev.jsonl`、`/tmp/harness_shared_data/train_ood.jsonl`、`/tmp/harness_shared_data/train_mcq.jsonl`，调用 `MyHarness` 内部检索和 prompt 构造函数，不调用 `call_llm`。
- 指标：DEV train 排名前三为 `recall24_e2_compact` selected 91.3%、候选均值 24.8、examples 2.0、prompt 1010/1144/1217；`recall20_e2_compact` selected 88.7%、候选均值 20.8、examples 2.0、prompt 873/1004/1060；`recall20_e4` selected 88.7%、prompt 1296/1523/1597。OOD train 排名前三为 `recall24_e2_compact` selected 94.1%、prompt 912/1047/1102；`recall20_e2_compact` selected 93.5%、prompt 792/927/982；`recall20_e4` selected 93.5%、prompt 1213/1462/1578。权重偏置不稳定：`bm25_heavy` 在 DEV 降到 83.1%，在 OOD 到 93.0%；`char_heavy` 在 DEV 87.9%，OOD 91.4%；`label_heavy` 在 DEV 87.9%，OOD 91.9%。
- 分析：LOO 的代理目标偏向“更多候选 + 更少 examples + 更短 label cards”。这在候选覆盖和 token 上看起来合理，但它会增加候选内判别难度；我们之前已经验证过离线候选召回提升并不必然提升 API accuracy。更稳妥的观察是 `compact_current`：不改变候选和 examples，只压缩 label cards，能显著降低 token，适合作为私有集 label 多或长文本时的预算预案，但它是否保持准确率仍需 API 验证。
- 决策：不落入 `solution.py`。不实现自动选择离线最优 preset，也不把 `recall24_e2_compact` 写死。若后续继续做 API 验证，只建议测试 1-2 个保守候选：`recall20_e2_compact` 用于检验候选覆盖收益，`compact_current` 用于检验 token 压缩是否不伤准确率；避免围绕 DEV 反复扩大搜索。
- 影响文件：`迭代决策记录.md`。
- 下一步：除非用户明确要求 API 验证，否则保持当前代码不变。若后续正式考虑 Auto-Tuning，必须把逻辑设计为提交时基于当前任务 train 自适应，而不是使用公开 DEV test 选出的固定参数。

### 2026-05-08 - 训练集内 Auto-Tuning 保守 API 验证

- 背景/触发：用户要求继续做 API 单轮验证两个保守候选，并追问为什么不做完整 API 验证。前置判断是：完整验证会把 DEV test 变成参数搜索目标，且历史已经证明离线召回提高可能导致 API 降分；因此先只验证两个机制上最保守的候选。
- 决策/改动：先确认本地 HEAD 与 GitHub 远端 HEAD 一致，当前基线已推送；随后用内存子类临时覆盖 `_route_task()` 和 `_format_label_cards()`，不修改 `solution.py`。`compact_current` 保持当前候选数和 example 数，只把 label cards 压短；`recall20_e2_compact` 使用 20 个候选、2 条 examples 和短 label cards，验证 LOO 中较保守的候选覆盖收益。
- 原因：`compact_current` 检验“token 压缩是否不伤准确率”，适合私有集 label 更多或文本更长时作为预算预案；`recall20_e2_compact` 检验“适度提高候选覆盖是否能弥补少 examples”。二者都比 `recall24_e2_compact` 更保守，能降低把 DEV 当宽网格刷分的风险。
- 预期效果：若 `compact_current` 与当前 80.9% 基本持平且 token 明显下降，可作为条件降级候选；若 `recall20_e2_compact` 超过 80.9%，再考虑是否需要 4 轮确认。若两者单轮都低于当前主线，则不继续扩大 API 网格。
- 实际效果：`compact_current` 官方 DEV 单轮 80.148%，432/539，低于当前 80.9%；prompt/条 781，completion/条 3.874，总耗时 59.8s，无截断、无 API 错误。`recall20_e2_compact` 官方 DEV 单轮 79.221%，427/539，低于当前 80.9%；prompt/条 864，completion/条 3.853，总耗时 57.2s，无截断、无 API 错误。
- 验证命令：`DASHSCOPE_API_KEY=<redacted> DASHSCOPE_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1 DASHSCOPE_MODEL=qwen3-8b LLM_RETRIES=0 LLM_RETRY_BASE_SLEEP=0.2 LLM_TIMEOUT=20 LLM_MAX_TOKENS=64 python -c <autotuning_conservative_api_validation_script>`；训练集为 `data/train_dev.jsonl`，验证集为 `data/test_dev.jsonl`，`workers=4`，每个候选单轮。
- 指标：当前落地主线 DEV 单轮为 80.9%，436/539，prompt/条 1074。`compact_current` 相比主线少 4 个正确样本，prompt 降约 27%；`recall20_e2_compact` 相比主线少 9 个正确样本，prompt 降约 20%。两者 `truncated=0`、`errors=0`。
- 分析：这次失败不是 API 调用或 token 预算问题，而是策略损失。`compact_current` 说明 label card 里的代表样本和关键词虽然占 token，但对候选内判别有帮助；过度压缩会丢细节。`recall20_e2_compact` 说明更高 LOO 候选覆盖仍会让 Qwen3-8B 面对更多相近 label，且 2 条 examples 不足以补偿判别难度。因此 LOO 可用于发现结构风险和预算预案，不能作为直接选参依据。
- 决策：不修改 `solution.py`，不继续做完整 Auto-Tuning API 网格，不把 `compact_current` 或 `recall20_e2_compact` 作为默认策略。当前主线仍是 label memory cards + 4 条 evidence examples + 隔离式 MCQ 路由 + 泛化 MCQ label 精确映射。
- 影响文件：`迭代决策记录.md`、`BASELINE_IMPLEMENTATION_PLAN.md`、`AGENT.md`。
- 下一步：只有当提交前风险检查发现私有集 prompt 接近 2048、label 数明显增大或出现截断风险时，再把 `compact_current` 作为条件降级方案重新评估；否则停止 Auto-Tuning API 搜索，避免围绕公开 DEV 过拟合。
