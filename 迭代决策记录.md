# 迭代决策与效果记录

本文档用于记录每次改进、调参、验证或方向切换的原因和实际效果。它服务于后续复盘和探索报告，不替代 `BASELINE_IMPLEMENTATION_PLAN.md`。

## 任务与目标

本次考核要求实现一个限制输入窗口的文本分类 Harness。系统先通过 `update(text, label)` 逐条输入带标签训练样本，随后通过 `predict(text)` 对测试样本返回一个 label 字符串。模型权重不更新，所有“学习”都必须发生在 Harness 维护的外部状态中，例如记忆、索引、检索、上下文压缩、输出解析和 fallback。最终返回值必须与训练集中出现过的 label 完全一致。

### 考核任务简介

本任务不是训练一个分类器，也不是把全部训练样本直接塞进 prompt，而是在 Qwen3-8B Instruct 非思考模式和单轮输入 2048 token 限制下，设计一个能泛化到多种私有任务的 Harness。私有评测包括与 DEV 相近的客服意图分类、领域和标签完全不同的 OOD 分类、复杂自然语言选择题，以及少量 prompt injection 样本。因此，方案既要追求准确率，也要避免只针对公开 DEV 集刷分。

### 输入输出接口

| 接口 | 输入 | 输出 | Harness 职责 |
| --- | --- | --- | --- |
| `update(text, label)` | 一条训练文本和对应 label | 无返回值 | 保存训练样本，维护 label allowlist，延迟构建检索索引和 label memory |
| `predict(text)` | 一条无标签测试文本 | 一个 label 字符串 | 路由任务类型，检索候选，构造预算内 prompt，调用 LLM，解析并校验输出 |

运行环境对 Harness 的关键含义是：预测阶段不能访问测试标签，也不能依赖外部文件或本地缓存；所有可用信息都应来自当前任务的训练流和 `predict()` 输入文本。

### 约束条件

| 约束 | 对设计的影响 |
| --- | --- |
| 只提交 `solution.py` | 最终策略必须完全封装在 `MyHarness` 内，不依赖本地临时脚本或外部数据文件 |
| 禁止读写文件 | 不能在提交逻辑中读取共享数据、FAQ、缓存、测试集或任何磁盘文件 |
| 测试 label 不可见 | 不能使用测试集答案或公开 DEV 错误样本做硬编码修正 |
| 单次输入 token 小于 2048 | 必须主动做候选裁剪、example 削减、长文本截断和预算检查 |
| 输出 exact match | 必须维护训练 label allowlist，并在 LLM 输出后做解析、归一化和 fallback |
| 私有集含 OOD/MCQ/injection | 不能只写 DEV 客服领域规则；需要任务路由、防注入边界和选择题专用处理 |
| 正式模型为 Qwen3-8B 非思考模式 | Prompt 不能过长、过复杂或依赖多步自由推理；应让代码承担确定性工作 |

### 评价维度

本报告用五类指标统一比较所有策略：

| 维度 | 关注点 | 记录方式 |
| --- | --- | --- |
| 准确率 | DEV、共享 MCQ/OOD、临时格式变体上的 exact-match accuracy | 单轮或 4 轮准确率、正确数 |
| 稳定性 | 多轮采样波动、API 错误、非法输出、并发下是否异常 | 4 轮均值、`api_error`、`invalid_output`、并发审计 |
| 泛化性 | 是否适用于 DEV-like、OOD、MCQ、不同 label 格式和 prompt injection | 外部分布验证、触发条件、是否避免 DEV 硬编码 |
| 预算控制 | prompt 是否接近或超过 2048，是否依赖自动截断 | prompt/条、p95、max、`truncated=0` |
| 提交合规性 | 是否只依赖 `solution.py`，是否无文件读写、无 API key、无测试标签泄漏 | 静态扫描、提交前风险审计 |

因此，一个策略不会仅因为单轮 DEV 略高就自动落入最终方案。只有当它同时满足收益明确、机制通用、预算安全、调用稳定和提交合规时，才会进入 `solution.py`。

### 初始 Baseline 定义

本报告中的 baseline 指最初建立的“增强版检索分类 Harness”：在 `update()` 中保存训练样本，在 `predict()` 中用 BM25、字符 n-gram 和 label 名称相似度检索候选 label，选择少量相关 examples 构造 prompt，让 Qwen3-8B 在受控候选集合中返回一个 label，并通过 allowlist parser 和本地 fallback 保证输出合法。

它被选为 baseline 的原因是：

- 它已经覆盖文本分类 Harness 的基础能力：外部记忆、动态检索、候选裁剪、few-shot、token 预算、防注入提示、输出解析和 fallback。
- 它没有引入高成本多轮调用，也不依赖 DEV 领域硬编码。
- 初始百炼单轮 DEV 为 80.5%，prompt/条约 451 token，无 API 错误、无截断 warning，说明基础链路稳定。

后续所有策略演化都以这个 baseline 为起点：先诊断它的错误来源，再决定是扩候选、改 prompt、加 label memory、做 MCQ 路由，还是停止某些高风险方向。

## 核心原则

- 先保证可回退：每次修改前确认当前本地基线已推送到 GitHub；没有稳定远端基线时，不继续做可能影响准确率的改动。
- 只让确定性代码做确定的事：记忆管理、检索、候选裁剪、token 预算、输出解析、fallback 和防注入由 Harness 控制；LLM 只在受控候选集合内做一次语义判断。
- 不为了 DEV 分数堆复杂度：只有实验收益明确、机制通用、不会明显增加限流和 token 风险时，才考虑把变体落入 `solution.py`。
- 负向实验也算结论：单轮显著低于当前基线区间的变体不再追加 4 轮确认；只有超过基线或提交前稳定性检查才跑多轮。
- 不把 API key、临时脚本或本地诊断逻辑写进提交文件；命令记录必须脱敏。
- 输出必须可控：最终返回必须来自训练 label allowlist；prompt injection 文本只作为待分类数据，不作为指令。
- 避免私有集风险：优先保留通用 Harness 机制，避免手写 DEV 客服领域规则、过度拟合错误样本或用检索覆盖模型判断。
- 默认单次调用：不默认启用 self-consistency、pairwise tournament 或多 agent；额外调用只在低置信、非法输出或正式风险明确时作为条件触发备选。

## 主线推进逻辑

先建立通用稳定基线，因为提交文件需要的不只是 DEV 准确率，还包括 token 预算、输出合法性、防注入、并发安全和 fallback。基线单轮 80.5%，且没有 API 或截断问题，所以后续不应先修基础设施，而应先诊断错误来源。

调用链路错误分型显示主要错误不是调用失败，也不是 parser 或 token 超限，而是候选命中后模型仍选错，以及少量候选 miss。因此先验证“候选更多、示例更多、规则更多、检索更强势”是否能解决问题。

候选扩张与规则调参的结果说明，离线召回提升不能直接转化为线上准确率；对 Qwen3-8B 来说，更大的候选集、更宽泛的规则和检索 override 都会带来新误判。于是探索范围收敛到更局部的 contrast / confusion-aware 小实验，并且默认单轮验证。

局部提示与轻量混淆对比仍然降分，说明当前瓶颈不能靠简单增加 label hints、共享 token 对比或示例重排解决。继续沿这个方向细刷 DEV，收益不确定，过拟合和 prompt 噪声风险更高。

Listwise 候选重排说明，把同一候选集包装成 ranking 任务不会自动提升候选内判别。Label Prototype / Concept Memory 的后续探索则给出了一个更具体的正向组合：不是用 prototype 替代 examples，也不是给候选行加少量 cues，而是用代码生成的 label memory cards 支撑更多候选，同时保留少量原始 examples。这个组合把 prompt 从 451/条提高到约 1074/条，但仍在 2048 预算内，并在 4 轮确认中稳定达到 80.891%。因此当前主线从“纯 few-shot 检索分类”升级为“候选检索 + label memory cards + 少量 evidence examples”的单调用 Harness。

共享数据侧增强审核把探索从 DEV 内部调参转到外部分布风险检查。`data.zip` 中新增 MCQ 和 OOD 数据没有文本重复或格式问题，适合作为“选择题长文本”和“非银行多域意图”的泛化验证集；但它们的 label 空间与 DEV 不同，不能作为当前任务的普通训练增强。直接把共享 train 合并到 DEV 会扩大 allowlist 并污染候选集，实际单轮也从 80.9% 降到 80.1%。只合并同名 `exchange_rate` 的 3 条样本虽然单轮到 81.1%，但收益仅约 1 个样本，且来自共享外部样本，不符合提交时不读外部文件、不过度依赖特定共享数据的原则。因此共享数据的结论是“用于验证和写报告，不进入提交 Harness”。

隔离式 MCQ 路由是在共享数据审核后的针对性增强：不把外部 MCQ 数据并入训练记忆，而是把 MCQ 作为一种可识别任务形态处理。实验显示，选择题 label `A/B/C/D` 本身没有语义，当前通用 few-shot 分类会把训练 examples 当作类别证据引入噪声；专用无示例 prompt 直接让 LLM 阅读题干和选项，成本更低且更准。该分支只在 choice label set 和 choice-like text 同时满足时触发，失效后回退通用 Harness，因此不会影响普通 DEV 路径。

FAQ 进一步说明，正式选择题任务是独立任务、单选、通用领域，但 label 不保证固定为 `A/B/C/D`，也不保证都是单字符；输出必须与训练集中 label 字符串完全一致。这使得单字母 MCQ 分支虽然安全，但覆盖不足。泛化 MCQ label 映射的策略是分层而不是替换：单字符 A-H 继续使用已验证有效的低噪声 prompt；非单字符选择题 label 只在能从 label 或题干选项中建立可靠映射时触发，并将模型输出的 `A`、`B`、`1` 等 marker 精确映射回训练集中原始 label 字符串。

训练集内 Auto-Tuning 离线 LOO 用来评估“提交时按当前任务 train 自适应”的可能性，而不是用公开 DEV test 刷分。结果显示，train 内候选覆盖确实可以通过 `recall20/24 + 更少 examples + 短 label cards` 提高，同时降低 prompt token；但这只说明 gold label 更常出现在候选集中，并不能证明 Qwen3-8B 在更大候选集里会选对。由于此前 `top16 + 12 examples` 已证明离线召回提升可能转化为线上降分，本方向暂不落代码，只作为后续少量 API 验证候选。

#### 架构含义说明

这张图表达的是一个“代码负责控制问题规模，LLM 负责小范围语义选择”的 Harness。训练阶段只积累外部记忆，不训练模型、不读写额外文件；预测阶段根据任务形态选择路径，并在调用 LLM 前完成候选裁剪、上下文压缩和 token 预算。

最终架构可以拆成 6 层：

| 层级 | 作用 | 关键机制 |
| --- | --- | --- |
| 记忆层 | 保存训练流信息 | `self.memory`、label allowlist、label 顺序 |
| 索引层 | 支持动态检索 | BM25、char n-gram、label-name similarity、label memory cards |
| 路由层 | 判断任务形态 | MCQ、小 label、普通多 label、prompt injection |
| 上下文层 | 组装最小充分 prompt | candidates、label cards、evidence examples、data boundary |
| 预算层 | 保证单次输入小于 2048 | 先减 examples，再缩候选，再截断长文本 |
| 输出层 | 保证 exact-match | parser、allowlist guardrail、retrieval fallback |

#### 设计取舍标注

这些内容不一定都画进主图，可以放在图旁或下一页：

- 不是纯 few-shot：全量训练样本塞不进 2048，而且会增加噪声；最终使用动态检索和候选裁剪。
- 不是纯传统分类器：私有集包含 OOD 和自然语言选择题，需要 LLM 的语义理解；传统检索只做候选生成和 fallback。
- 不是多轮投票或 tournament：调用成本、延迟和限流风险高，且错误分型没有显示多次调用是主要瓶颈。
- MCQ 独立处理：选项 label 本身没有类别语义，训练 examples 反而干扰；隔离式低噪声 prompt 更适合选择题。
- 保留 label memory cards：prototype 不能替代 examples，但能在更多候选下提供压缩后的类别概念，最终与少量 evidence examples 组合取得稳定正向结果。

| 共享数据独立验证 | 当前 Harness 跑共享 MCQ/OOD | MCQ 73.2%；OOD 78.0% | 作为验证集 | 数据本身有泛化验证价值，但不进入训练增强 |
| 隔离式 MCQ 路由 | A-H label + choice-like text 走无示例选择题 prompt | MCQ 75.5%；DEV 80.9%；prompt/条从 MCQ 628 降到 189 | 是 | MCQ label 无类别语义，隔离低噪声读题能提升选择题且不影响普通分类 |
| 泛化 MCQ label 映射 | `Option A/B/C/D`、`A. xxx`、数字 marker 到原 label | 原始 MCQ 保持 75.5%；`Option A/B/C/D` 78.6%；DEV 80.9% | 是 | 支持正式可能出现的非单字符 label，保证 exact-match 返回训练 label |
| Train-only Auto-Tuning LOO | train 内 leave-one-out preset 网格 | `recall24_e2_compact` selected hit 在 DEV/OOD train 为 91.3%/94.1% | 不落代码 | 只作为代理诊断，不能证明 LLM 最终会选对 |
| Auto-Tuning API 验证 | `compact_current`、`recall20_e2_compact` | DEV 80.148%、79.221%，均低于当前 80.9% | 否 | token 降低和 LOO 覆盖收益没有转化为 API accuracy |
| 提交前风险审计 | 静态扫描 + 无 API 预算/并发压力审计 | DEV max 1496、OOD max 1463、MCQ max 695；无非法返回、无超预算、无合规问题 | 是，作为提交依据 | 当前 `solution.py` 满足提交边界，剩余风险主要是未知私有格式 |

这张表的结论可以在报告中写成：

> 最终方案不是单纯选择 DEV 分数最高的局部变体，而是选择了同时满足准确率、泛化性、预算安全、调用稳定和提交合规的机制组合：Label Memory + 少量 evidence examples + 隔离式 MCQ 路由 + 精确 label 映射 + guardrail/fallback。

#### 失败案例 / 失败方向分析表

| 失败方向 | 表面动机 | 实际现象 | 得到的结论 | 对最终方案的影响 |
| --- | --- | --- | --- | --- |
| 更多候选和 examples | 提高 gold label 召回，让模型看到更多证据 | `top16 + 12 examples` 候选命中率提升，但 DEV 降到 79.6% | 召回不是唯一瓶颈；候选内判别会被更多相似 label 和 examples 干扰 | 最终只适度扩大候选，并用 label memory cards 压缩概念，不盲目堆 examples |
| Prompt 规则加法 | 用规则帮助模型区分相近 label | DEV 降到 78.7% | Qwen3-8B 对宽泛自然语言规则敏感，规则可能变成噪声 | Prompt 保持短、直接、任务边界清晰，不继续堆通用判别规则 |
| 检索 top-1 override | 当检索高置信时纠正 LLM 错误 | 阈值网格无组合超过 80.5% 基线 | 检索高分不等于语义正确，本地检索不能替代语义判断 | 检索只做候选生成和非法输出 fallback，不覆盖合法 LLM 输出 |
| 局部 hints / contrast | 给相近 label 额外差异提示 | label hints 78.5%，contrast 79.2%，example order 78.5% | 轻量局部信息没有稳定提升，反而增加上下文噪声 | 不做手写 confusion 规则，避免 DEV 领域过拟合 |
| Listwise reranking | 让模型对候选排序，缩小开放分类难度 | 同候选 ranking 也降到 77.7%，候选-only 68.6% | 排序格式和枚举比较对小模型更难；label 名称本身不足以表达类别语义 | 最终保留直接分类 prompt，不把任务改成 ranking |
| 共享数据直接合并 | 增加训练信息，提升泛化 | DEV 降到 80.1%；外部 label 进入大量 DEV 候选 | 不同任务 label 空间混合会污染 allowlist 和候选集 | 共享数据只做外部分布验证，不进入提交 Harness |
| 仅同名外部样本增强 | 用少量同名 label 样本增强概念 | `exchange_rate` 3 条增强 DEV 到 81.1%，但收益约 1 个样本且证据薄 | 小样本偶然收益不能作为提交逻辑，且存在硬编码/数据依赖风险 | 不读取外部文件，不硬编码共享样本 |
| 复杂 MCQ prompt | 显式解析选项或加入每选项 examples | `parse_options` 71.35%，`one_per_option_example` 70.57% | MCQ 需要低噪声读题，训练 examples 跨题干反而干扰 | 最终 MCQ 分支采用无示例 prompt |
| 所有 MCQ 统一泛化 prompt | 用一套完整 label prompt 覆盖所有选择题 label 格式 | 原始单字母 MCQ 从 75.5% 降到 72.1% | 泛化能力不能以伤害已验证路径为代价 | 分层实现：单字符 A-H 保留旧路径，非单字符才启用映射 |
| Auto-Tuning 默认化 | 用 train LOO 自动选更优 preset | LOO 代理指标提升，但 API 验证 `compact_current` 80.148%、`recall20_e2_compact` 79.221% | 离线候选覆盖代理不能直接代表 LLM 最终 accuracy | 不把 LOO 最优 preset 写死，只保留 token 压力预案 |
| Self-consistency / Pairwise 默认启用 | 多次调用投票或两两比较提升稳定性 | 未默认跑完整实验；基于成本和已有错误分型判断不划算 | 调用次数会放大延迟、限流和随机性，且当前瓶颈不证明需要多次调用 | 最终保持默认单次调用，仅作为理论备选 |

这张表的结论可以在报告中写成：

> 多数失败方向有共同特征：它们试图通过“更多上下文、更复杂 prompt、更多候选或更多调用”提升表现，但在 Qwen3-8B 和 2048 token 约束下，额外信息经常转化为噪声。最终方案因此收敛到低噪声、单调用、强约束的 Harness：代码负责缩小问题和校验输出，LLM 只做最后的受控语义选择。

### 4. 最终方案与总结

这一节用于最终报告的结论页。它不再展开每个实验细节，而是回答三个问题：最终提交的 Harness 是什么，为什么没有把所有看起来有用的东西都堆上去，以及这轮探索得到的方法论和局限性是什么。

#### 最终方案一句话

最终提交方案是一个单调用、强约束、检索增强的分类 Harness：

> 代码侧负责外部记忆、混合检索、候选裁剪、label memory cards、任务路由、token 预算、输出解析和 fallback；Qwen3-8B 只在受控候选集合内做一次最终语义选择。

最终 `solution.py` 中保留的核心机制：

| 模块 | 最终采用机制 | 作用 |
| --- | --- | --- |
| 外部记忆 | `update()` 保存训练样本、label 顺序和 allowlist | 保证输出空间来自当前任务训练流 |
| 混合检索 | BM25 + char n-gram + label-name similarity | 在 2048 token 内缩小候选 label 空间 |
| Label Memory | 代码生成 label cards + 少量 evidence examples | 用压缩类别概念支撑更多候选，同时保留原始样本证据 |
| 任务路由 | 普通分类、小 label、MCQ、injection-aware 分支 | 让不同任务形态使用不同上下文策略 |
| MCQ 路由 | 无示例选择题 prompt + marker 到原 label 映射 | 降低选择题噪声，支持非单字符 label exact-match |
| 预算控制 | 调用前削减 examples、候选和长文本 | 主动保证 prompt 小于 2048，不依赖自动截断 |
| 输出控制 | parser + allowlist guardrail + retrieval fallback | API 异常或非法输出时仍返回训练 label |

#### 为什么不把所有高分或有用信号都堆上去

最终方案选择的是“通用、可解释、可提交”的机制组合，而不是把所有单轮看起来有收益的东西全部合并。

最典型的例子是共享数据中只合并同名 `exchange_rate` 的 3 条外部样本后，DEV 单轮到 81.1%。这个结果没有进入最终方案，原因是：

- 收益只有约 1 个样本，证据薄，不能证明稳定泛化。
- 样本来自外部共享文件，正式提交不能读外部文件，硬编码也会违反 Harness 自适应叙事。
- 它只增强一个 DEV 相关 label，可能提高公开 DEV，却降低私有 OOD/MCQ 的可信度。
- 这类做法会把报告从“任务内自适应 Harness”拉向“数据侧补丁”，复核风险更高。

同样，Train-only Auto-Tuning 的 LOO 代理指标看起来更好，但两个保守 API 候选都低于当前主线；因此它只保留为诊断信号和 token 压力预案，不默认落代码。最终采用标准是：一个机制必须同时满足机制通用、实验收益明确、不增加明显 token/调用风险、不会依赖外部数据、不会污染 label 空间。

#### 最终结果摘要

| 维度 | 最终状态 |
| --- | --- |
| DEV 准确率 | 当前主线单轮 80.9%，Label Memory 正向组合临时 4 轮均为 80.891% |
| MCQ 泛化 | 隔离式 MCQ 路由后 MCQ 75.5%；非单字符 `Option A/B/C/D` label 格式 78.6% |
| OOD 验证 | 当前 Harness 在共享 OOD 独立任务单轮 78.0%，作为外部分布参考 |
| Token 预算 | DEV max 1496，OOD max 1463，MCQ max 695，均低于 2048 |
| 调用稳定性 | 已有主线验证无 API 错误、无 prompt truncated warning |
| 提交合规 | `solution.py` 无文件读写、无 API key、无网络调用、无 DEV 测试集硬编码 |

#### 已知局限性

| 局限 | 具体风险 | 当前缓解方式 |
| --- | --- | --- |
| 私有格式未知 | 正式 MCQ label 或题干格式可能不同于共享样本 | 只在可建立可靠 marker 映射时触发 MCQ，否则回退通用分类 |
| 极端长文本 / 小 label 任务 | 小 label 且长文本时 prompt 可能接近预算上限 | 预算循环会削减 examples、候选和文本；审计中 8 类长文本 max 1916 |
| Qwen3-8B 候选内判别有限 | 候选命中后仍可能选错，错误分型显示这是主要瓶颈 | 用 label memory cards + 少量 evidence examples 降低判别难度，但不强行多轮投票 |
| 检索不是语义真值 | BM25/char/label-name 分数高不一定等于正确 label | 检索只用于候选生成和 fallback，不覆盖合法 LLM 输出 |
| Label memory 压缩可能丢细节 | 过短 cards 会降低准确率，`compact_current` API 低于主线 | 当前保留代表样本片段和 cues，不启用过度压缩默认策略 |
| 不默认多次调用 | self-consistency/pairwise 可能在少量难例上有用 | 出于时间、限流和稳定性考虑，最终保持单调用，只作为理论备选 |

这些局限不说明当前方案不完整，而是说明最终方案选择了保守边界：在无法看到私有测试集的前提下，优先提交机制通用、调用稳定、预算安全、可解释的 Harness。

#### 探索方法论

本轮探索形成了三条方法论，可以作为报告总结的核心反思：

1. **离线代理指标只能诊断，不能直接替代 API 验证。**
   候选命中率、train-only LOO selected hit、prompt token 都有价值，但它们不等于最终 exact-match accuracy。`top16 + 12 examples` 和 Auto-Tuning LOO 都证明，覆盖率提高可能带来候选内判别噪声。

2. **负向实验缩小了搜索空间。**
   扩候选、prompt 规则、检索 override、局部 hints、listwise、共享数据合并和 Auto-Tuning preset 的失败，让最终方案更有说服力。它们说明不是所有“更多信息”都会帮助小模型，很多时候更多上下文就是更多噪声。

3. **Harness 的价值在于限制不确定性。**
   最终方案把不确定性限制在一次受控 LLM 判断中：代码负责记忆、检索、路由、预算和校验；LLM 只在候选集合内选 label。这比让 LLM 自由读长 prompt、多轮比较或开放输出更适合 Qwen3-8B、2048 token 和 exact-match 评分环境。

#### 报告结论可直接使用版本

最终报告可以用下面这段作为收束：

> 最终提交的 Harness 不是把所有实验中看起来有效的技巧相加，而是在多轮正负实验后保留机制通用、预算安全、调用稳定、可解释且符合提交约束的部分。实验证明，简单增加候选、示例、规则或外部数据并不稳定，甚至会降低准确率。最终方案采用 label memory cards + 少量 evidence examples 提升普通分类，用隔离式 MCQ 路由和精确 label 映射增强选择题，用预算控制、allowlist parser 和 fallback 保证合规与稳定。这个设计的核心思想是：Harness 负责把任务缩小和校验，LLM 只做一次受控语义选择。
