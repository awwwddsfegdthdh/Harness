# 迭代决策与效果记录

本文档用于记录每次改进、调参、验证或方向切换的原因和实际效果。它服务于后续复盘和探索报告，不替代 `BASELINE_IMPLEMENTATION_PLAN.md`。

## 核心原则

- 先保证可回退：每次修改前确认当前本地基线已推送到 GitHub；没有稳定远端基线时，不继续做可能影响准确率的改动。
- 只让确定性代码做确定的事：记忆管理、检索、候选裁剪、token 预算、输出解析、fallback 和防注入由 Harness 控制；LLM 只在受控候选集合内做一次语义判断。
- 不为了 DEV 分数堆复杂度：只有实验收益明确、机制通用、不会明显增加限流和 token 风险时，才考虑把变体落入 `solution.py`。
- 负向实验也算结论：单轮显著低于当前基线区间的变体不再追加 4 轮确认；只有超过基线或提交前稳定性检查才跑多轮。
- 不把 API key、临时脚本或本地诊断逻辑写进提交文件；命令记录必须脱敏。
- 输出必须可控：最终返回必须来自训练 label allowlist；prompt injection 文本只作为待分类数据，不作为指令。
- 避免私有集风险：优先保留通用 Harness 机制，避免手写 DEV 客服领域规则、过度拟合错误样本或用检索覆盖模型判断。
- 默认单次调用：不默认启用 self-consistency、pairwise tournament 或多 agent；额外调用只在低置信、非法输出或正式风险明确时作为条件触发备选。

## 阶段间决策逻辑

第一阶段先建立通用稳定基线，因为提交文件需要的不只是 DEV 准确率，还包括 token 预算、输出合法性、防注入、并发安全和 fallback。基线单轮 80.5%，且没有 API 或截断问题，所以后续不应先修基础设施，而应先诊断错误来源。

阶段 1.5 显示主要错误不是调用失败，也不是 parser 或 token 超限，而是候选命中后模型仍选错，以及少量候选 miss。因此第二阶段先验证“候选更多、示例更多、规则更多、检索更强势”是否能解决问题。

第二阶段结果说明，离线召回提升不能直接转化为线上准确率；对 Qwen3-8B 来说，更大的候选集、更宽泛的规则和检索 override 都会带来新误判。于是第三阶段只允许更局部的 contrast / confusion-aware 小实验，并且默认单轮验证。

第三阶段三个局部变体仍然降分，说明当前瓶颈不能靠简单增加 label hints、共享 token 对比或示例重排解决。继续沿这个方向细刷 DEV，收益不确定，过拟合和 prompt 噪声风险更高。

因此第四阶段不再默认实现 label prototype / concept memory。这个方向理论上适合 token 紧张、label 很多或长文本任务，但当前 prompt 平均 451、p95 543、最大约 728，离 2048 限制很远；在没有 token 压力时引入 prototype，反而可能压缩掉有用样本细节并增加实现复杂度。现阶段更合理的目标是保留稳定基线，完成提交前风险检查和探索报告，把负向实验作为“为什么不继续改”的证据。
